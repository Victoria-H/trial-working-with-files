Job Documentation

Overview

Having just spent some time on the Planning phase, this section will quickly cover important best practices related to the Development phase as it pertains to documentation. Specifically, several guidelines with respect to Job properties, Component names and labels as well as using notes in the Design Workspace are introduced. Once that is completed, you will progress to the Documentation phase and generate and view HTML documentation for a Job you built earlier. As with any project, good documentation can be critical for other people to understand a Talend Job.

Improve Key Job Properties

  1. BE AS CLEAR AS POSSIBLE ABOUT THE PURPOSE AND DESCRIPTION OF JOBS

    Open the WeatherWebService Job and then click the Job view :

    This is the information you supplied when you created the Job. In the interest of time, not much effort was spent on the Purpose or Description fields of this and other Jobs. That is not considered best practice, so it should get fixed.

  2. REFINE YOUR TEXT FOR THE JOB PURPOSE

    Enter something more helpful for the Purpose field. Such as Get weather by ZIP code.

  3. REFINE THE JOB DESCRIPTION

    Similarly, enter something more helpful for the Description field. Such as Send a ZIP code to a public SOAP web service which returns the current weather.

  4. SAVE THE JOB

    Save the changes to your Job.

    Note that you can also change both of these Job properties without even opening the Job. Right-click the Job in the repository and click Edit properties:

Document a Component

Thus far, you have placed Components in the Design Workspace and left their default names alone, and provided little to no further information. This section will introduce you to several best practice ideas surrounding Component names and other helpful descriptive information.

  1. GIVE FURTHER INFORMATION ABOUT COMPONENTS

    Open the Component view for tESBConsumer. Click the View tab:

  2. ENTER A DESCRIPTIVE LABEL

    Enter the following for the Label format: <b>__UNIQUE_NAME__</b><br>Invoke Get Weather method

    3. At a glance in the Design Workspace, you can see the component icon, name (in bold) and a brief reminder what it does. All of this information can be helpful, especially for new developers or others looking at your Job.

  3. ADD A HINT TO A COMPONENT

    Change the Hint format below the Label format to something more helpful such as: Invoke public SOAP weather service using the GetWeather method with a ZIP code. Note that the tESBConsumer component still does not reflect the change. (The next step will enable the Hint format field.)

  4. CHECK THE Show Information BOX

    Change to the Documentation view for the tESBConsumer component. Click the Show Information check box:

  5. ADD OTHER HELPFUL INFORMATION

    Enter other helpful information for someone to know about this component in the Comment field, such as: The full method (operation) name for the get request is GetCityWeatherByZIP. Response is GetCityWeatherByZIPResponse. Note that you will see this information when you view and generate the HTML documentation in the next section. (Since you are in the Documentation view now, it's convenient to enter the information now as well.)

  6. NOTICE THE INFORMATION ICON NEXT TO THE COMPONENT

    Hover over the "i" above the tESBConsumer component in the Design Workspace and notice the Hint format is displayed:

  7. SIMILARLY DOCUMENT OTHER COMPONENTS

    Time and interest permitting, repeat these steps to better document another component, such as one of the tXMLMap components.

  8. SAVE

    Save the Job.

View and Generate HTML Documentation

Before generating the documentation and viewing it from the file system, you will view it directly from within the Studio. This is a nice feature for quickly checking what the HTML looks like during the development cycle which may contain many iterations.

  1. VIEW HTML JOB DOCUMENTATION FROM THE STUDIO

    In the Repository, right-click WeatherWebService > View documentation:

    A new tab with an HTML documentation preview is opened up.

  2. EXAMINE THE COMPONENT LIST

    You will have more time to peruse the documentation shortly. For now, simply click the Component List hyperlink.

  3. CHECK THE DETAILS OF THE COMPONENT DOCUMENTED EARLIER

    Select tESBConsumer from the list of components in your Job. Notice the detailed Component Parameters as you scroll down. Towards the bottom of the parameters you should see the Comment Property and it's associated Value which matches the Comment you added earlier in this exercise:

    4. Now that viewing documentation from within the Studio looks correct, you can generate a more permanent version of the documentation.

  4. GENERATE HTML OUTPUT FOR THE JOB DOCUMENTATION

    Right-click the WeatherWebService Job in the Repository and then click Generate Doc As HTML:

  5. SELECT A LOCATION AND SAVE

    Browse and set the directory path to C:\StudentFiles, use the default File name and click Save when ready:

  6. THE HTML OUTPUT IS SAVED IN AN ARCHIVE FILE

    The Generate Documentation dialog should look similar to this:

     

  7. EXTRACT THE CONTENTS

    Locate the archive in the directory structure and then extract the contents using 7zip. (Hint: Right-click the file > 7-Zip > Extract here.)

    Locate and open the WeatherWebServices_0.1.html file below the WeatherWebService_0.1 folder. Spend a few minutes and examine the documentation created. You can see that all of your comments are included, as well as component configuration settings, and a variety of other data:
